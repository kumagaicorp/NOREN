# UI Spec Compliance Checker

実装がUX仕様に準拠しているかを検査する。

## 入力

- UX仕様書（docs/UX/）
- 実装コード（apps/desktop/src/, packages/ui/）
- 実装のスクリーンショット（任意）

## 出力

- 準拠チェック結果（Pass/Fail/Warn）
- 乖離箇所一覧
- 修正提案
- 承認 / 要修正 の判定

## 検査観点

### 1. レイアウト構造
- UX仕様のレイアウト構造と一致しているか
- 要素の配置が正しいか
- グループ化が適切か

### 2. 情報設計
- 優先度が仕様通りか
- 表示情報が過不足ないか
- 視覚的ヒエラルキーが正しいか

### 3. 操作フロー
- 仕様通りの操作が可能か
- フロー順序が正しいか
- 状態遷移が適切か

### 4. 状態表現
- 初期/ローディング/空/正常/エラーが実装されているか
- 各状態の表現が仕様通りか

### 5. a11y
- ラベルが設定されているか
- フォーカス管理が適切か
- キーボード操作が可能か
- ダイアログのフォーカストラップがあるか

### 6. tokens使用
- 色/余白/角丸がハードコードされていないか
- tokens を正しく使用しているか

### 7. 受入基準
- Given/When/Then が満たされているか

## 出力フォーマット

```markdown
## 準拠チェック結果

### 判定: [Pass / Fail / Warn]

### 乖離箇所一覧

| 観点 | 仕様 | 実装 | 判定 | 修正提案 |
|------|------|------|------|----------|
| レイアウト | ... | ... | Fail | ... |
| 情報設計 | ... | ... | Pass | - |

### 詳細
[各観点の詳細説明]

### 総評
[全体的なコメント]
```

## ワークフロー

1. 対象のUX仕様書を読む
2. 実装コードを確認する
3. 各検査観点でチェックする
4. 乖離箇所を特定し、一覧化する
5. 修正提案を具体的に記述する
6. 承認 / 要修正 の判定を下す

## 受入条件

- UX仕様書を全て確認している
- 7つの検査観点を全て確認している
- 乖離箇所が具体的に特定されている
- 修正提案が実装可能なレベルで具体的
