# PRD

## 背景 / 目的
- まずは MVP を A（ローカルHTTP）で構築し、早期に動作確認できる状態を作る
- PRD 確定後に要件へ合わせて接続方式を再選定する
- MVP は「最小の動作確認」と「開発導線の確立」を主目的とする

## ユーザーストーリー
- 開発者として、Desktop から Backend を呼び出せる最小経路を確認したい
- 開発者として、起動・停止・疎通確認が短時間でできる状態にしたい

## 機能要件
- Desktop から Backend をローカル HTTP で呼び出せる
- Backend は `/health` を提供し、200 を返す
- Desktop 側は疎通確認を1画面で確認できる
- 画面には「起動状態」「疎通結果」「再試行」操作がある

## 非機能要件
- 早期に起動・実行できる最小構成
- ローカル開発環境で再現可能な手順がある

## MVP 画面/操作フロー
- 画面起動時に Backend の `/health` を呼び出す
- 成功時は「接続OK」を表示する
- 失敗時は「接続NG」と原因メッセージを表示する
- 「再試行」ボタンで再度 `/health` を呼び出す

## 受入基準（MVP）
- Desktop 起動後、初回の疎通結果が画面に表示される
- `/health` が 200 のとき「接続OK」が表示される
- `/health` が失敗したとき「接続NG」とエラーメッセージが表示される
- 「再試行」を押すと結果が更新される

## 制約
- PRD 未確定のため、要件変更を前提にする
- 本番配布や署名は対象外（MVP の範囲外）
